This document specifies the standard constraints on the OSLC Source Control Management (SCM) and OSLC Configuration Management vocabulary terms using OSLC ResourceShapes. The SCM vocabulary defines domain-specific resource types for source control artifacts (files, directories, changes), while Configuration Management types (Stream, Baseline, ChangeSet, Component) are defined in the OSLC Configuration Management namespace [[OSLCConfig]].

Introduction

RDF vocabularies define the terms and resources for a domain of interest, source control management in the case of OSLC SCM. These vocabularies are often specified in an open manner, without providing information such as property domain and range assertions, cardinalities, etc. This helps keep the vocabulary applicable for a wide range of uses and furthering integration with other vocabularies.

However, it is often desirable to close down a vocabulary with specific constraints to facilitate using the vocabulary for a specific purpose. This document specifies the constraints for using the OSLC SCM vocabulary and relevant OSLC Configuration Management vocabulary terms in an OSLC SCM server backed by GitHub. Different sets of constraints may be applied to a vocabulary in order to tailor its use, without overly constraining the vocabulary for other usages.

These constraints apply to the SCM domain vocabulary defined in OSLC SCM Version 1.0. Part 1: Vocabulary and the OSLC Configuration Management vocabulary defined in [[OSLCConfig]].

Terminology

Terminology is based on OSLC Core Overview [[!OSLCCore3]], W3C Linked Data Platform [[!LDP]], W3C's Architecture of the World Wide Web [[WEBARCH]], Hyper-text Transfer Protocol [[!HTTP11]].

Concept Resource
An artifact as an overall entity, independent of any specific version. It has a stable URI used for linking. See [[OSLCConfig]].
Version Resource
A specific state of a concept resource at a point in time. Version resources are resolved from concept resources using a Configuration Context. See [[OSLCConfig]].
Configuration
Identifies a set of versions of resources in a component. Streams are mutable configurations; baselines are immutable. See [[OSLCConfig]].
Component
A unit of organization consisting of a set of versioned resources. In the GitHub context, a component corresponds to a GitHub repository.

References

In addition to the namespace URIs and namespace prefixes oslc, rdf, dcterms and foaf defined in the OSLC Core specification, OSLC SCM defines the namespace URI of http://open-services.net/ns/scm# with a namespace prefix of oslc_scm.

This specification also uses these namespace prefix definitions:

  • oslc_config : http://open-services.net/ns/config# [[OSLCConfig]]

Configuration Management Resource Constraints

This section specifies the constraints for Configuration Management resources used in an SCM context. These resource types are defined in the OSLC Configuration Management namespace [[OSLCConfig]] and are included here with SCM-specific constraints for the GitHub server implementation.

Resource: Component

A Component represents a unit of organization consisting of a set of versioned resources. In the GitHub context, a component corresponds to a GitHub repository.

Resource: Stream

A Stream is a mutable configuration representing a line of development. In the GitHub context, a stream corresponds to a Git branch.

Resource: Baseline

A Baseline is an immutable configuration representing a fixed snapshot of versioned resources. In the GitHub context, a baseline corresponds to a Git tag.

Resource: ChangeSet

A ChangeSet represents a unit of change applied to a configuration. In the GitHub context, a changeset corresponds to a Git commit.

SCM Domain Resource Constraints

This section specifies the constraints for SCM domain-specific resources. These resource types are defined in the OSLC SCM namespace. The resource properties are not limited to the ones defined in this specification; SCM servers may provide additional properties. It is recommended that any additional properties exist in their own unique namespace and not use the namespaces defined in these specifications.

Resource: FileVersion

A FileVersion represents a specific version of a file under source control. FileVersion resources are versioned resources that have both a concept URI (identifying the file across all versions) and a version URI (identifying a specific state at a branch, tag, or commit). In the GitHub context, a file version corresponds to a blob at a specific Git ref.

Resource: DirectoryVersion

A DirectoryVersion represents a specific version of a directory under source control. In the GitHub context, a directory version corresponds to a tree at a specific Git ref.

Resource: Change

A Change represents a single modification within a change set. Changes are typically inlined within their containing ChangeSet resource rather than existing as independent resources. In the GitHub context, a change corresponds to a file diff within a commit.

Resource: BaselineComparison

A BaselineComparison represents the difference between two baselines. This is an on-demand resource computed by comparing two Git tags.

Resource: FileVersionComparison

A FileVersionComparison represents the difference between two file versions, including an optional unified diff.

Resource: DirectoryVersionComparison

A DirectoryVersionComparison represents the difference between two directory versions, showing members added and removed.

Conformance

SCM servers MUST follow the constraints defined here where required, and with the meanings defined here.

SCM servers MAY provide additional constraints for specific purposes.
