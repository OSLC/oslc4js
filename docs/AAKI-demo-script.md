# AAKI in 10 minutes

## A live demo of Define, Instantiate, Activate against `bmm-server`

A scripted walkthrough that makes the AAKI value proposition concrete. One server, three stages, three artifacts on screen, and a closing-the-loop analysis the audience can verify by URI.

> *Companion slide deck:* this is the **demo script**. The fuller framework deck is `AAKI-Presentation.md`; the worked-example deep-dive is `AAKI-Presentation-Example.md`. Use *this* deck when an audience needs to *see* AAKI work, not read about it.

# Beat 1 — Define

## Show what the AI authored before today

**On screen:** open `BMM-Shapes.html` in the browser.

**What to point at (in order):**

1. **The TOC** — 14 shapes, one per instantiable BMM class.
2. **GoalShape** — expand it. Read off two or three properties (`amplifiedBy`, `quantifiedBy`, `madeOperativeBy`).
3. **The "Inverse" column** — *"Amplifies"*, *"Quantifies"*, *"Made Operative By"*. This is what makes incoming-link discovery clean — no hardcoded inverse table on any client.
4. **Predicate names** — short verb phrases. Note that none of them encode the target class (`amplifiedBy`, not `amplifiedByMission`).

<div class="pn">

**Say it like this:** *"This was authored by Claude reading the OMG BMM 1.3 specification — the same PDF you can download from omg.org. About 25 classes, 49 properties, 14 shapes, validated by the OSLC-OP ShapeChecker. The whole vocabulary fits in three Turtle files plus this HTML rendering — no domain-specific application code anywhere."*

</div>

<!-- _class: beat small-text -->

# Beat 1 — what just landed

A complete OSLC service contract for a non-trivial domain, **without** a developer writing one line of domain code:

- **`BMM.ttl`** — the open vocabulary (classes and properties as plain identifiers, no `rdfs:domain`/`rdfs:range`).
- **`BMM-Shapes.ttl`** — the API contract: cardinalities, value types, ranges, inverse metadata.
- **`BMM-Shapes.html`** — the human-browsable reference.
- **`config/catalog-template.ttl`** — generated by `create-oslc-server.ts`, declaring the creation factories, dialogs, and query capability for each managed class.
- **The running `bmm-server`** — `oslc-service` middleware on Apache Jena Fuseki, serving the OSLC + MCP + LDM endpoints automatically.

> *The shape **is** the contract. Every operational surface — REST API, browser UI, LDM `/discover-links`, MCP tool schemas — is generated from it.*

# Beat 2 — Instantiate

## Populate EU-Rent live, via MCP

**On screen:** Claude Desktop (left); browser at `http://localhost:3005/` (right), empty `eu-rent-demo` ServiceProvider.

**Paste into Claude Desktop:**

```
Populate the EU-Rent example (BMM 1.3 Annex C) onto bmm-server at
http://localhost:3005/mcp. First read_catalog, then get_resource on
the shapes you'll need, then create the ServiceProvider and ~72
linked resources across all BMM types. Forward links only; report
progress every 10.
```

**Audience sees:** (1) `read_catalog` and `get_resource` on shape URIs — the discover-first protocol in action; (2) a burst of `create_Vision`, `create_Goal`, `create_Strategy`, `create_Tactic`, … — each a typed, shape-validated POST; (3) the browser tab populating in near-real-time as the AI works.

<div class="pn">

**Fallback if MCP is slow:** kill the prompt mid-run, switch to a terminal, run `./bmm-server/testing/populate-eurent.sh`, explain *"same MCP calls, scripted for replay — same outcome, ~60 seconds."*

</div>

# Beat 2 — what just landed

**~72 linked, governed resources** authored from a single prompt:

| Type | Count | What was created |
|---|---|---|
| Vision / Mission | 1 + 1 | The EU-Rent vision and the mission that makes it operative |
| Goal / Objective | 4 + 4 | Premium positioning, customer service, fleet upkeep, availability |
| Strategy / Tactic | 3 + 5 | Channelling effort toward goals; tactics implementing each strategy |
| Policy / Rule | 5 + 6 | Governance directives + concrete enforcement rules |
| Influencer | ~20 | External (regulators, competitors, technology) and internal (capabilities, assumptions) |
| Assessment / Impact | 6 + 5 | SWOT-style assessments and the impacts they identify |
| Process / Asset / OrgUnit | 4 + 4 + 4 | Realization scaffolding |

Three things to point at: **every resource attributed to the user** (`dcterms:creator` is the human, not the AI); **every link is typed** (queryable, not text-similarity); **the AI never strayed outside the shapes** (validation rejected the only attempt it made).

# Beat 3 — Activate

## Ask the graph a question that needs the AI

**On screen:** Claude Desktop. Browser tab still showing the populated graph for visual reference.

**Paste this prompt:**

```
Identify Goals on the EU-Rent ServiceProvider that have no realizing
Tactic chain. The expected chain is Goal ← channelsEffortsToward — Strategy
← implements — Tactic. For each gap, give the Goal's title and URI, the
nearest existing coverage, and what kinds of Tactics would close the gap.
Cite every Goal by URI. Don't create anything — just report.
```

**What the audience sees (in order):**

1. The discover-first preamble — Claude reading the catalog, the GoalShape, the StrategyShape, the TacticShape.
2. Three or four `query_resources` calls fanning out across the graph.
3. A two- or three-row table naming Goals **with URIs**, the nearest coverage they have, and concrete suggestions for closing each gap — grounded in the spec's text, not invented.

<div class="pn">

**Audience cue:** open the gap Goal in the oslc-browser (column 1 → click the Goal → no incoming Strategy in column 2). The browser shows the same gap the AI found. *That's verifiability — the AI's claim is checkable in seconds.*

</div>

# Beat 3 — what just landed

The AI did **not** carry hardcoded BMM knowledge. It read the shapes to learn the chain, queried the data, and joined the layers to identify missing edges.

Five things made the answer trustworthy:

1. **Every claim cited a resource URI** — so the audience can navigate to it.
2. **Predicates were used verbatim** from the shapes — `channelsEffortsToward`, not "drives" or "supports".
3. **The data was authoritative** — the AI queried the live graph, not a cached snapshot.
4. **Gaps were explicit** — when a chain ended early, the AI said so rather than papering over it.
5. **No writes happened** — the prompt was pure analysis. To write, the same pattern goes through Observe-Propose-Execute (slide below).

> *Same archetype works for: impact analysis ("if this requirement changes, what's affected?"), coverage reports ("which hazards lack mitigations?"), compliance checks ("every OrgUnit must be responsible for at least one End"), and AI-drafted resources for human review.*

# What's actually being demonstrated (one slide)

| AAKI stage | What the audience saw | Without AAKI, this would be... |
|---|---|---|
| **Define** | A full OSLC contract for BMM, authored from the spec PDF, validated by ShapeChecker | Weeks of hand-coding by a senior modeler + integration developer |
| **Instantiate** | 72 linked, shape-conformant resources from a single MCP-driven prompt | A multi-day SME engagement; in practice, the model never gets populated |
| **Activate** | A gap-analysis answer with cited URIs in seconds, replayable for impact / coverage / compliance / proposed-resource patterns | A bespoke report someone hand-writes, manually verified, often stale before it's read |

The pattern is **domain-neutral**. Replace BMM with **ISO 9001**, **MRM**, **RM**, **QM**, or any RDF-expressible ontology — same prompts, same skills, same outcome.

# The RACI line — what the AI never does

Throughout the demo, the AI:

- Used **the user's credentials** to act against the server. Every change is attributable to the human.
- Operated inside **the user's chosen working context** (a scratch ServiceProvider for this session) — never created configurations / change sets / branches on the user's behalf.
- Never **delivered**, **merged**, **promoted**, or **published** anything. Those are human actions.
- For any ambiguous proposal, surfaced the **Observe-Propose-Execute** pattern: drafts a resource, stops, waits for explicit approval before calling `create_*`.

> *AI assistants in AAKI are **collaborators, not agents on the RACI chart**. Humans remain Responsible and Accountable; the governance trail proves the human owned the outcome.*
